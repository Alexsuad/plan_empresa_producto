# AGENTS.md — Reglas del proyecto plan_empresa_producto

## 1. Identidad del proyecto

Este repositorio no es todavía un sistema técnico en producción.

Este repositorio contiene la base documental y funcional para diseñar un futuro sistema agéntico que acompañe a emprendedores en la creación de un Plan de Empresa profesional, validado, trazable y defendible.

## 2. Alcance actual

La fase actual es de producto, documentación y arquitectura funcional.

Está permitido trabajar sobre:

- documentos de producto;
- arquitectura funcional;
- mapas de roles;
- gates;
- requisitos de skills;
- matrices de consistencia;
- estructura de preinformes;
- blueprint funcional.

No está permitido crear todavía:

- backend;
- frontend;
- base de datos;
- agentes ejecutables;
- integraciones MCP;
- automatizaciones productivas;
- despliegues;
- código del sistema final.

## 3. Idioma y estilo

Toda comunicación, documentación, planes y explicaciones deben estar en español claro, sencillo y profesional.

El lenguaje debe ser comprensible para una persona que está aprendiendo programación y arquitectura de sistemas.

## 4. Regla de trabajo

Trabajar siempre así:

1. Diagnosticar.
2. Proponer.
3. Esperar aprobación cuando el cambio sea estructural.
4. Ejecutar solo lo aprobado.
5. Verificar con evidencia.
6. Cerrar con resumen.

## 5. Reglas de seguridad

No borrar archivos sin aprobación explícita.

No modificar `.env`, secretos, claves, tokens o configuraciones sensibles.

No ejecutar comandos destructivos.

No introducir código productivo sin autorización.

No convertir este repositorio en una app de formularios ni en una herramienta solo de marketing.

## 6. Trazabilidad

Todo documento nuevo debe tener propósito claro, ubicación correcta y relación con la fase actual.

Si se cambia una decisión relevante, debe registrarse en documentación de gobierno o arquitectura funcional.

## 7. Uso de IA y determinismo

La IA puede proponer, explicar, resumir, detectar inconsistencias y redactar borradores.

Las tareas mecánicas, repetibles o sensibles deben validarse con terminal, Git, scripts o comandos deterministas.

## 8. Cierre obligatorio

Todo trabajo debe cerrar indicando:

- archivos creados o modificados;
- resumen de cambios;
- verificación realizada;
- estado final;
- próximos pasos sugeridos.
